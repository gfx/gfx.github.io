889=head1 DESCRIPTION
890
891B<DBD::mysql> is the Perl5 Database Interface driver for the MySQL
892database. In other words: DBD::mysql is an interface between the Perl
893programming language and the MySQL programming API that comes with
894the MySQL relational database management system. Most functions
895provided by this programming API are supported. Some rarely used
896functions are missing, mainly because noone ever requested
897them. :-)
898
899In what follows we first discuss the use of DBD::mysql,
900because this is what you will need the most. For installation, see the
901sections on L<INSTALLATION>, and L<WIN32 INSTALLATION>
902below. See L<EXAMPLE> for a simple example above.
903
904From perl you activate the interface with the statement
905
906 use DBI;
907
908After that you can connect to multiple MySQL database servers
909and send multiple queries to any of them via a simple object oriented
910interface. Two types of objects are available: database handles and
911statement handles. Perl returns a database handle to the connect
912method like so:
913
914 $dbh = DBI->connect("DBI:mysql:database=$db;host=$host",
915 $user, $password, {RaiseError => 1});
916
917Once you have connected to a database, you can can execute SQL
918statements with:
919
920 my $query = sprintf("INSERT INTO foo VALUES (%d, %s)",
921 $number, $dbh->quote("name"));
922 $dbh->do($query);
923
924See L<DBI(3)> for details on the quote and do methods. An alternative
925approach is
926
927 $dbh->do("INSERT INTO foo VALUES (?, ?)", undef,
928 $number, $name);
929
930in which case the quote method is executed automatically. See also
931the bind_param method in L<DBI(3)>. See L<DATABASE HANDLES> below
932for more details on database handles.
933
934If you want to retrieve results, you need to create a so-called
935statement handle with:
936
937 $sth = $dbh->prepare("SELECT * FROM $table");
938 $sth->execute();
939
940This statement handle can be used for multiple things. First of all
941you can retreive a row of data:
942
943 my $row = $sth->fetchrow_hashref();
944
945If your table has columns ID and NAME, then $row will be hash ref with
946keys ID and NAME. See L<STATEMENT HANDLES> below for more details on
947statement handles.
948
949But now for a more formal approach:

1452=head1 STATEMENT HANDLES
1453
1454The statement handles of DBD::mysql support a number
1455of attributes. You access these by using, for example,
1456
1457 my $numFields = $sth->{'NUM_OF_FIELDS'};
1458
1459Note, that most attributes are valid only after a successfull I<execute>.
1460An C<undef> value will returned in that case. The most important exception
1461is the C<mysql_use_result> attribute: This forces the driver to use
1462mysql_use_result rather than mysql_store_result. The former is faster
1463and less memory consuming, but tends to block other processes. (That's why
1464mysql_store_result is the default.)
1465
1466To set the C<mysql_use_result> attribute, use either of the following:
1467
1468 my $sth = $dbh->prepare("QUERY", { "mysql_use_result" => 1});
1469
1470or
1471

1475Column dependent attributes, for example I<NAME>, the column names,
1476are returned as a reference to an array. The array indices are
1477corresponding to the indices of the arrays returned by I<fetchrow>
1478and similar methods. For example the following code will print a
1479header of table names together with all rows:
1480

1486 die "Error:" . $sth->errstr . "\n";

1488 my $names = $sth->{'NAME'};
1489 my $numFields = $sth->{'NUM_OF_FIELDS'};
1490 for (my $i = 0; $i < $numFields; $i++) {
1491 printf("%s%s", $i ? "," : "", $$names[$i]);
1492 }
1493 print "\n";
1494 while (my $ref = $sth->fetchrow_arrayref) {
1495 for (my $i = 0; $i < $numFields; $i++) {
1496 printf("%s%s", $i ? "," : "", $$ref[$i]);

1623=head1 TRANSACTION SUPPORT
1624
1625Beginning with DBD::mysql 2.0416, transactions are supported.
1626The transaction support works as follows:
1627
1628=over
1629
1630=item *
1631
1632By default AutoCommit mode is on, following the DBI specifications.
1633
1634=item *
1635
1636If you execute
1637
1638 $dbh->{'AutoCommit'} = 0;
1639
1640or
1641
1642 $dbh->{'AutoCommit'} = 1;
1643
1644then the driver will set the MySQL server variable autocommit to 0 or
16451, respectively. Switching from 0 to 1 will also issue a COMMIT,
1646following the DBI specifications.
1647
1648=item *
1649
1650The methods
1651
1652 $dbh->rollback();
1653 $dbh->commit();
1654
1655will issue the commands COMMIT and ROLLBACK, respectively. A
1656ROLLBACK will also be issued if AutoCommit mode is off and the
1657database handles DESTROY method is called. Again, this is following
1658the DBI specifications.
1659
1660=back
1661
1662Given the above, you should note the following:
1663
1664=over
1665
1666=item *
1667
1668You should never change the server variable autocommit manually,
1669unless you are ignoring DBI's transaction support.
1670
1671=item *
1672
1673Switching AutoCommit mode from on to off or vice versa may fail.
1674You should always check for errors, when changing AutoCommit mode.
1675The suggested way of doing so is using the DBI flag RaiseError.
1676If you don't like RaiseError, you have to use code like the
1677following:
1678
1679 $dbh->{'AutoCommit'} = 0;
1680 if ($dbh->{'AutoCommit'}) {
1681 # An error occurred!
1682 }
1683
1684=item *
1685
1686If you detect an error while changing the AutoCommit mode, you
1687should no longer use the database handle. In other words, you
1688should disconnect and reconnect again, because the transaction
1689mode is unpredictable. Alternatively you may verify the transaction
1690mode by checking the value of the server variable autocommit.
1691However, such behaviour isn't portable.
1692
1693=item *
1694
1695DBD::mysql has a "reconnect" feature that handles the so-called
1696MySQL "morning bug": If the server has disconnected, most probably
1697due to a timeout, then by default the driver will reconnect and
1698attempt to execute the same SQL statement again. However, this
1699behaviour is disabled when AutoCommit is off: Otherwise the
1700transaction state would be completely unpredictable after a
1701reconnect.
1702
1703=item *
1704
1705The "reconnect" feature of DBD::mysql can be toggled by using the
1706L<mysql_auto_reconnect> attribute. This behaviour should be turned off
1707in code that uses LOCK TABLE because if the database server time out
1708and DBD::mysql reconnect, table locks will be lost without any
1709indication of such loss.

1776=head1 MULTITHREADING
1777
1778The multithreading capabilities of DBD::mysql depend completely
1779on the underlying C libraries: The modules are working with handle data
1780only, no global variables are accessed or (to the best of my knowledge)
1781thread unsafe functions are called. Thus DBD::mysql is believed
1782to be completely thread safe, if the C libraries are thread safe
1783and you don't share handles among threads.
1784
1785The obvious question is: Are the C libraries thread safe?
1786In the case of MySQL the answer is "mostly" and, in theory, you should
1787be able to get a "yes", if the C library is compiled for being thread
1788safe (By default it isn't.) by passing the option -with-thread-safe-client
1789to configure. See the section on I<How to make a threadsafe client> in
1790the manual.

1792=head1 ASYNCHRONOUS QUERIES
1793
1794You can make a single asynchronous query per MySQL connection; this allows
1795you to submit a long-running query to the server and have an event loop
1796inform you when it's ready. An asynchronous query is started by either
1797setting the 'async' attribute to a truthy value in the L<DBI/do> method,
1798or in the L<DBI/prepare> method. Statements created with 'async' set to
1799true in prepare always run their queries asynchronously when L<DBI/execute>
1800is called. The driver also offers three additional methods:
1801C<mysql_async_result>, C<mysql_async_ready>, and C<mysql_fd>.
1802C<mysql_async_result> returns what do or execute would have; that is, the
1803number of rows affected. C<mysql_async_ready> returns true if
1804C<mysql_async_result> will not block, and zero otherwise. They both return
1805C<undef> if that handle is not currently running an asynchronous query.
1806C<mysql_fd> returns the file descriptor number for the MySQL connection; you
1807can use this in an event loop.
1808
1809Here's an example of how to use the asynchronous query interface:
1810
1811 use feature 'say';
1812 $dbh->do('SELECT SLEEP(10)', { async => 1 });
1813 until($dbh->mysql_async_ready) {
1814 say 'not ready yet!';
1815 sleep 1;
1816 }
1817 my $rows = $dbh->mysql_async_result;
